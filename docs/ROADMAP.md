# ROADMAP.md — 開発計画・タスクバックログ

AI開発憲法における「タスク仕様書」に相当するが、個人開発の規模に合わせて
1ファイルに集約している（1タスク1ファイルにする程の規模ではないため）。
タスク完了時、仕様や設計に関わる決定があればDECISION_LOG.mdにADRを追記する。

## 進め方の基本方針

Claudeは実機のレンダリング結果を直接確認できない。そのため、見た目や操作感に
関わるタスクは「Claudeが実装 → ユーザーが実機(iPhone 14 Safari)で確認 →
フィードバック → Claudeが修正」のループを前提に進める。憶測で完璧を狙わず、
まず動くものを素早く出して実機で検証するサイクルを優先する。

## A. フィードバック待ち（ユーザー側アクション待ち）

- [ ] ポーズ4種（直立/ピース/手を振る/腰に手）の実機調整結果(JSON)待ち
      → 受領後 `main.js` の該当 `bones` 値へ反映し、DECISION_LOG.mdへ追記
- [ ] 輪郭線・トゥーンの見た目についての感想待ち（ADR-004参照）
      → 「MMD感が強い」場合はPBRシェーディングへの切り替えを検討
- [ ] ライト構成の是正(ADR-006)後、「白っぽさ・異物感」が改善したかの確認待ち
      → まだ白っぽい場合は`renderer.toneMappingExposure`をさらに下げる、
        または`HemisphereLight`をさらに弱める/撤去する方向で追加調整
- [ ] 手ぶれ補正(ADR-007)の効き具合の確認待ち
      → AR固定のガクつきが減ったか／シャッター時の「安定待ち」の時間感覚が
        ちょうど良いか(待たされすぎ/短すぎ)。`STEADY_THRESHOLD`等の閾値を実機に合わせて調整
- [ ] 追加したいポーズの要望確認（振り返り／顎に手 など）

## B. 次にやる想定のタスク

1. [ ] 受領したポーズJSONを `main.js` に反映し、動作確認手順を更新
2. [ ] PWA対応（`manifest.json`、ホーム画面アイコン、`apple-touch-icon`）
       → Flowly / VocaDesk と作法を揃える
3. [ ] 2人目のMMDモデル追加
   - 3-1. PMX解析（テクスチャ参照名・モーフ名・ボーン名の確認、命名ズレの吸収）
   - 3-2. `expressions` / `poses` を2人目モデル用に個別定義（モーフ名はモデル固有のため使い回し不可）
   - 3-3. キャラクター選択画面が2枚目のカードを正しく出すか動作確認
4. [ ] 環境光推定によるモデルの陰影・色味の自動調整（優先度低のため保留していた項目）
5. [ ] 2D透過素材キャラクターの実装検証（実際の素材が用意でき次第）

## C. 保留・要検討（着手条件が整うまで様子見）

## 運用ルール（このプロジェクトでの開発サイクル）

1. Claude側で実装 → zipで納品
2. ユーザーが実機(iPhone 14 Safari)で動作確認
3. 不具合・違和感をフィードバック（ポーズ関連はポーズ調整モードのJSON出力を活用すると早い）
4. Claude側で修正。設計判断を伴う場合はDECISION_LOG.mdにADRを追記
5. スコープ自体を拡げる・変える大きな変更をする時は、着手前にCONSTRAINTS.mdを更新する
