# 推しカメラ (oshi-camera) 開発引き継ぎ書

対象: `oshi-camera/` リポジトリ一式（GitHub Pagesで運用、Flowly/VocaDeskとは別リポジトリ）

推しカメラは、iPhone(Safari)のカメラ映像にMMDモデルを重ねて、その場で配置・ポーズ・表情を
調整してから撮影するAR風の個人利用アプリです。ビルド不要（Three.jsをCDN読み込み）で
GitHub Pagesにそのまま置ける構成です。

**このファイルは「次のチャットで最初に読む1枚」です。** 個々の詳細は下記ドキュメントに
分散しているので、状況を掴んだらそちらを参照してください。

## 1. 最重要の前提（何度も繰り返し確認してきたこと）

- **Claudeはこのサンドボックス環境から、ブラウザ実機でのレンダリング結果を直接確認できない。**
  Chromiumはsnap配布のみでこの環境では動かせず、ヘッドレスレンダリングも不可（試して断念済み、DECISION_LOG ADR-006参照）。
  → 見た目・操作感に関わる数値（ポーズの関節角度、ライトの強さ、エフェクトの強度等）は
    **すべて「実機で見ずに作ったたたき台」**である前提で扱うこと。憶測で完璧を狙わず、
    「Claude実装→ユーザーが実機(iPhone 14 Safari)で確認→フィードバック→Claude修正」の
    ループを回す。
- **ユーザーは現在、実機での動作確認が一時的にできない状態**（最新の依頼時点）。そのため
  直近の作業は「後で検証すればいい形で機能の幅を広げる」方向にシフトしている。
  実機確認が再開できたかどうかは会話の最初で確認するとよい。
- 見た目の不具合は、**憶測で直さず可能な範囲でThree.js等のライブラリのソースコードを
  直接取得して確認してから直す**、というのがこのプロジェクトで確立した手法（ADR-006, ADR-008）。
  `raw.githubusercontent.com` からthree.jsの該当バージョンのソースを取得できる。
- **実機写真のフィードバックを初めて受け取った（ADR-010）。** 輪郭線・色味の白飛びは
  改善が確認できた一方、ポーズ(ピース/手を振る)の指・手首の角度が意図通りに見えていない
  ことが分かった。ユーザー自身も「どのポーズを押したか忘れた」状態だったため、
  ポーズ/表情選択時にトースト表示する機能を追加した。次回は選択直後の画面を見ながら
  ポーズ調整モードを使ってもらう想定。

## 2. 現在実装済みの機能（ざっくり一覧）

- カメラ映像＋MMDモデル(天音かなた、ファンメイド)のAR風合成・撮影
- 1本指ドラッグ移動／2本指ピンチ拡縮・回転
- ジャイロによる3DoF AR固定（向きのみ、位置追従はできない。ADR-002, ADR-007）
- 表情プリセット（通常/笑顔/ウインク/びっくり/困り）＋自動まばたき＋微小ランダム変動
- ポーズプリセット9種（直立/ピース/手を振る/腰に手/考え中/座る/振り返り/両手ピース/指ハート）
  ＋アプリ内「ポーズ調整モード」（実機でスライダー調整→JSON出力→Claudeに送る運用）
- 手ぶれ補正（AR固定の減衰スラープ＋撮影時の静止待ち、ADR-007）
- セルフタイマー／構図グリッド／アングル切替（全身・上半身・顔アップ、カメラのドリー方式）
- 環境光推定（平均色・輝度・簡易光源方向・露出、ADR-008, ADR-009）
- Shadow Quality（二重影＋接触AO＋光源方向オフセット）
- 輪郭線を黒からトーンカラーへ、トゥーン階調をLinearFilterで滑らかに
- 撮影後処理：ビネット・フィルムグレイン・軽い色収差・疑似ブルーム・カラーグレーディング
  （彩度/コントラスト/トーンカーブ/環境色ティント、ADR-010）
  （**撮影結果のみ、ライブプレビューには非適用**）
- ポーズ/表情選択時に画面上部へ一瞬名前を表示するトースト通知（ADR-010、「どのポーズか忘れる」対策）
- 遠近法：カメラFOVを前面/背面で個別に設定（実機レンズ諸元の概算根拠あり）
- 動画撮影機能（**実装済みだがユーザー指示により一時保留中**、mode-btnで写真/動画切替、MediaRecorder使用）

## 5. 壊しやすい箇所・ハマりどころ

- **PMXのテクスチャファイル名に注意**: `tex_kanata_face parts.png` / `tex_kanata_short pants.png`
  はPMX内部の参照名が半角スペース区切りなのに対し、元のアップロードファイル名は
  アンダースコア区切り。`assets/kanata/tex_kanata/` にはPMXが期待する名前（スペース区切り）で
  配置してあるので、テクスチャを差し替える際はこの命名ルールに注意。
- **HemisphereLightはMMDのトゥーン階調を経由せずベタ塗りで加算される**（three.jsのMMDToonShaderソースで確認済み、ADR-006）。強くすると全体が白っぽく平坦になる。DirectionalLightは階調を経由するので、そちらを強めにするのが基本。
- **`preserveDrawingBuffer: true`が無いと撮影時にモデルが消える**（WebGLの描画バッファが既定で保存されないため、ADR-003）。
- **`touch-action: none` + `preventDefault()`が無いとiOSが2本指ジェスチャーをページズームとして奪う**（ADR-003）。
- **`-webkit-touch-callout: none`を安易に全体へかけると、結果画像の長押し保存メニューまで消える**（ADR-003）。操作エリアだけに範囲を限定すること。
- **`getFootY()`はSkinnedMeshの静的なbind-poseバウンディングボックスを使っており、ボーンの変形(座りポーズ等)を厳密には反映しない**（ADR-008）。センターオフセット分だけ既知の補正を入れているが、脚の曲げによる正確な接地位置までは反映できていない。
- **ポーズ/表情バーのレイアウトは`#ui-layer`の`justify-content:space-between`が絡む**。以前ここに複数の子要素を直接置いて画面中央に来てしまうバグを起こした（ADR-008で修正済み）。UI要素を追加する時は`#top-controls`と`#bottom-controls`の中に入れること。
- **撮影後処理(postfx.js)はライブプレビューには反映されない、撮影結果にのみ適用**という設計上の判断がある（ADR-009）。「プレビューと違う」というフィードバックが来ても仕様通りなので、必要なら設計変更を検討する。
- **ボーン名・モーフ名はモデル固有**。2人目のモデルを追加する時は`expressions`/`poses`/`idle`をすべて個別に定義し直す必要がある（使い回し不可）。PMXバイナリを解析してテクスチャ参照名・モーフ名・ボーン名を確認するPythonスクリプトの手法は本チャットのログに残っている（`pymeshio`は追加UVチャンネルに非対応で使えなかったため、自前でPMXバイナリパーサーを書いた）。

## 6. 著作権・利用範囲について

天音かなたのファンメイドMMDモデルを使用しているため、**公開・販売はしない**方針で
ユーザーと合意済み（CONSTRAINTS.md参照）。「商業アプリ並みの品質」という要望は
あくまで個人利用の範囲内での品質・UX向上を指す。
